# 原功能重写取舍规划

最后更新：2026-06-27

这份文档用于回答一个核心问题：旧 Writingway 的功能还要不要继续逐项重写。

结论：不应该逐项照搬。这个 fork 已经转向桌面端、本地优先、项目目录存储和原生工作台体验。旧功能只作为参考实现；是否重写，取决于它是否能服务新的桌面产品结构。

## 判断原则

- 保留能直接支撑写作闭环的功能：项目、编辑、生成、阅读、备份、恢复、资料管理。
- 重做理解成本高但价值明确的功能：AI 设置、提示词、上下文、资料库、workshop。
- 弱化旧 Web 时代的重复入口：多处弹窗、滑出面板、堆叠按钮、同一能力的多个入口。
- 删除或归档没有清晰桌面价值的旧实现：未加载模块、重复适配层、只服务旧 UI 的中间状态。
- 半自动小说工作流最后做，只在其他功能稳定后进入正式设计和实现。

## 已经由新主线接管

- 桌面入口和主导航。
- 作品库/书架。
- 项目目录存储。
- 原生项目编辑器。
- 原生阅读器。
- 原生生成面板。
- 原生备份和恢复中心。
- 核心项目模型、文稿组装、阅读转换。
- generation core、prompt/result/history 基础。
- 工作流占位 schema 和事件日志占位。
- 测试分组、打包、发布检查。

## 仍需重写的高价值旧功能

### AI 设置和 Provider 配置

旧设置页功能很全，但混在 `main.html` 和旧 Alpine 状态里。新主线需要一个更克制的原生设置页：

- provider 切换。
- API key、base URL、model 配置。
- OpenAI-compatible 自定义 provider。
- LM Studio/Ollama/local endpoint 配置。
- 默认生成参数。
- 连接测试。
- 设置导入导出。

不建议照搬旧设置页的复杂布局。桌面端应该更像系统设置：分组明确、保存状态清楚、错误可诊断。

### 资料库 / 世界观

旧 compendium/worldbuilding 是有价值的，但 UI 和数据关系需要重做：

- 角色、地点、组织、物品、设定、时间线等类型。
- 项目目录内持久化。
- 与场景、章节、生成上下文关联。
- 支持搜索、标签、引用统计。
- 支持从编辑器和 workshop 快速引用。

这会成为工作流之前最重要的“知识层”。

### 上下文和提示词系统

旧 context panel 和 prompts 面板功能有价值，但应抽成更清晰的核心层：

- PromptTemplate：写作、改写、总结、workshop、工作流步骤。
- ContextResolver：从当前项目、场景、资料库、手动选择项构建上下文。
- PromptPreview：任何生成前都能查看最终请求。
- ContextBudget：控制上下文长度，避免 prompt 不可控。

这是未来半自动工作流能否稳定的关键前置条件。

### Workshop / 创作对话

旧 workshop 值得保留，但不应作为旧 iframe 的侧栏继续膨胀。新版本应变成原生创作助手：

- 按项目保存对话。
- 可引用场景、资料库条目、当前选中文本。
- 复用统一 provider 和 prompt/context core。
- 输出可一键转成资料库条目、场景摘要、章节想法或正文片段。

### 导入导出和迁移

导入导出应服务桌面项目目录，而不是只服务旧 JSON：

- 旧 JSON snapshot 导入。
- Writingway 1 项目导入。
- Markdown/TXT 导出。
- 项目目录备份包导出。
- 未来 EPUB 导入或导出可放到后续。

### 云端/扩展备份

GitHub Gist 备份存在旧实现，但不是桌面主线的刚需。建议排在本地备份稳定之后：

- 先保留旧实现作为兼容。
- 新主线优先保证本地备份、恢复、导入导出可靠。
- 云同步/远端备份作为独立扩展，不和核心编辑保存逻辑耦合。

## 可以弱化或归档的旧功能形态

- 旧 Web 时代的大量滑出 panel。
- 多套重复的生成入口。
- 与旧 IndexedDB 强绑定的项目状态。
- 未被 `main.html` 实际加载的历史模块。
- 旧 UI 内部的临时 prompt preview/debug 输出。
- 没有清晰桌面用户路径的“占位云服务”入口。

## 后续阶段路线

### 阶段 12：原生设置和 Provider 配置

目标：让新桌面主线不再依赖旧 AI 设置页。

产出：

- 原生设置页重做。
- settings service/storage。
- provider 配置 schema。
- provider 连接测试 API。
- 生成面板读取新设置。

### 阶段 13：原生资料库 / 世界观

目标：把旧 compendium/worldbuilding 迁移为项目目录内的知识层。

产出：

- 资料条目 schema。
- 项目目录存储。
- 原生资料库 UI。
- 编辑器和生成面板可引用资料条目。
- 旧 compendium 导入适配。

### 阶段 14：上下文和提示词系统

目标：把 prompt 和 context 从 UI 状态中彻底拿出来。

产出：

- PromptTemplate schema。
- ContextResolver core。
- prompt/context 存储。
- 原生 prompt 管理 UI。
- 所有生成入口统一走 prompt/context core。

### 阶段 15：原生 Workshop / 创作对话

目标：重做 workshop，让它成为桌面创作助手，而不是旧侧栏复刻。

产出：

- 对话存储。
- 原生 chat UI。
- 引用场景、资料库、选中文本。
- 对话输出转资料/摘要/草稿。

### 阶段 16：导入导出和兼容收束

目标：把旧数据入口变成明确的迁移能力。

产出：

- 旧 JSON snapshot 导入器完善。
- Writingway 1 导入器接入新项目目录。
- 项目目录打包导入/导出。
- Markdown/TXT 导出质量提升。
- 明确旧 IndexedDB 项目迁移路径。

### 阶段 17：旧 iframe 退出准备

目标：把旧 iframe 从“必要运行时”降级为“参考/兼容工具”。

产出：

- 列出 iframe 仍独有的功能。
- 对已迁移功能移除旧入口依赖。
- 清理未加载和重复模块。
- 将 legacy tests 缩小为迁移兼容测试。

### 阶段 18：正式半自动小说工作流

目标：在前面能力稳定后，才开始真正实现复杂 workflow。

前置条件：

- 原生编辑器稳定。
- 原生生成稳定。
- 资料库稳定。
- prompt/context 稳定。
- 备份/恢复稳定。
- 运行前快照和回滚策略稳定。

这一阶段不应提前开始。
