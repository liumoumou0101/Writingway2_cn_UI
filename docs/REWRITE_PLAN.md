# Writingway 后续重构与重写规划

最后更新：2026-06-27

## 目标

下一轮工作的目标不是继续给旧 Web 应用补功能，而是把 Writingway 逐步变成一个真正的桌面本地写作工作台。

核心方向：

- 新功能默认写在桌面主线。
- 业务逻辑默认写在 `src/core` 或 `desktop/services`。
- 旧 iframe writer 只保留为兼容运行时，逐步缩小职责。
- 半自动小说工作流最后做，不提前把复杂度引入主线。

## 当前基础

已经完成：

- 纯核心项目模型。
- 本地项目目录格式。
- 桌面 project service。
- 原生书架和项目目录识别。
- 原生编辑器 MVP。
- 原生阅读器核心转换。
- 原生恢复页 MVP。
- 生成基础契约。
- 工作流占位持久化。

这意味着后续可以开始做“产品化重写”，而不是继续做地基。

## 推荐顺序

### 1. 旧代码边界整理

先把旧代码职责写清楚，避免后面改错地方。

输出：

- `docs/LEGACY_BOUNDARY.md`
- legacy-only 测试分组
- desktop-mainline 测试分组

### 2. 原生编辑器产品化

目标是让用户可以不进入旧 iframe，就能完成日常写作。

重点：

- 章节/场景完整 CRUD。
- 排序。
- 自动保存。
- 元数据编辑。
- 搜索替换。
- 更好的键盘操作。

### 3. 原生生成界面迁移

目标是让 AI 生成从旧 Alpine UI 迁到桌面 writer。

重点：

- beat 输入。
- prompt 预览。
- streaming 生成。
- accept/retry/discard。
- reasoning 展示。
- generation history。

### 4. 原生备份恢复

目标是让用户敢用新编辑器和未来工作流。

重点：

- 备份预览。
- diff。
- 单场景恢复。
- 整项目恢复。
- 恢复为新项目。

### 5. 半自动小说工作流

最后开始正式实现。

第一版只做半自动，不做全自动：

```text
项目设定
  -> 生成章节大纲
  -> 用户确认
  -> 生成场景列表
  -> 用户确认
  -> 逐场景生成草稿
  -> 用户接受/重试/编辑
```

每一步都必须：

- 可暂停。
- 可重试。
- 可回放 prompt。
- 可查看输出。
- 可恢复到运行前快照。

## 不做的事

短期不做：

- 不重写成复杂前端框架。
- 不提前做全自动长篇生成。
- 不把工作流逻辑塞进 UI 按钮事件里。
- 不让 IndexedDB 重新成为主数据源。

## 完成标准

可以认为重写完成的标准：

- 新建、打开、编辑、保存、阅读、备份、恢复都不依赖旧 iframe。
- AI 生成主流程不依赖旧 `src/app.js`。
- 项目主数据只以本地项目目录为主源。
- 旧 JSON snapshot 只用于导入/导出/兼容。
- 半自动工作流有可恢复、可回放、可人工确认的最小闭环。
